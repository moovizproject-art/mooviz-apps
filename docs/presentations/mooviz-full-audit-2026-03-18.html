Security Audit

5 Critical + 8 High Severity Findings

Critical Findings

S1
Client writes status directly via Firestore rules — bypasses Cloud Functions state machine. A malicious client can set any status (e.g., jump new → completed_paid).
CRITICAL
S2
Client writes driverId — any sender can assign arbitrary users as drivers without consent.
CRITICAL
S3
Client writes interestedDrivers array — no validation on structure. Fake driver entries can be injected.
CRITICAL
S4
purgeTestUsers is unauthenticated HTTP endpoint — anyone can enumerate and delete all users.
CRITICAL
S5
13+ direct Firestore writes from mobile screens — status transitions, payment, proof fields written without server validation.
CRITICAL

High Severity Findings

S6KYC not enforced — assertDriverApproved defined but never called. Pending/rejected drivers can operate.
S7Role check too permissive — sender with driverUnlocked bypasses activeMode check.
S8Dual delivery creation paths — direct Firestore + Cloud Function. Direct path skips validation.
S9Ratings world-readable — allow read: if true exposes user IDs to unauthenticated users.
S10No rate limiting on email callable — bulk email function has no throttle.
S11User IDs in Cloud Function logs — violates "no PII in logs" policy.
S12Chat rules allow any field modification — participants can overwrite closed, participants, etc.
S13API keys in git history — google-services.json was previously committed. Keys should be rotated.

Priority Fix: Remove from Firestore Rules

Remove status, driverId, interestedDrivers from allowedSenderDeliveryFields and allowedDriverDeliveryFields. This closes S1 + S2 + S3 simultaneously. Delete purgeTestUsers endpoint (S4).

Migration Scripts Review

Glide → Firestore Data Migration

6 scripts reviewed: migrate-glide-data.ts, fix-glide-migration.ts, migrate-images.ts, validate-migration.ts, fix-itemsize.ts, fix-stuck-deliveries.ts

Critical Findings

C1
Field name mismatch — Migration writes item.description / item.size but app expects itemDescription / itemSize. Runtime fallbacks mask this. Firestore indexes on itemSize miss migrated records.
C2
Missing expansion fields — notifiedDrivers, notifyRadius, notifyExpansionCount, lastNotifyExpansion not set. Migrated "new" deliveries won't be picked up by notifyExpansion scheduler.
C3
Coordinate field mismatch — Migration writes lat/lng but app expects latitude/longitude. Map screens and distance calculations may fail.

High Severity Findings

H1No idempotency guard — migrateShipments uses .add(). Re-running creates duplicate deliveries. No glideId check.
H2statusHistory format mismatch — Migration writes array; app writes map (statusHistory.${status}: timestamp).
H3Message migration creates duplicates — subcollection add() on re-run.
H4Hardcoded credential paths — Scripts reference specific user's ~/.config/ directory.
H5Signed URLs expire 2030 — Stored in Firestore, will break in 4 years.
H6Default status fallback is completed_paid — Unrecognized Hebrew status silently maps to terminal state.

Medium & Low Findings

MEDIUMEmpty proof object — completed Glide deliveries lose proof photos
MEDIUMfix-itemsize.ts batch exceeds 500-op Firestore limit for large datasets
MEDIUMmakePublic() failure silently swallowed — returns 403 URL
LOWCSV parser doesn't handle UTF-8 BOM from Windows Excel exports
LOWcity field always empty on migrated deliveries
LOWtimeoutAt set to current time = immediately timed out

Existential Risks

Zero Monetization

No platform fee, no commission, no revenue path. The price goes directly from sender to driver via external apps. Interstitial ads are the only planned revenue — not yet implemented.

Payment Friction

4-6 extra steps vs competitors. External Bit/PayBox deep links, manual confirmation by both parties, screenshot upload. This will be the #1 user complaint.

Cold-Start Problem

Chicken-and-egg: need drivers in every area for senders to get responses. No referral program, no driver incentives, no guaranteed earnings. Geographic density is critical.

Strengths

  • Multi-driver interest system (sender picks their driver)
  • Proof photos for pickup, delivery, and payment
  • Bidirectional ratings with comments
  • Custom notification sounds per event (6 mapped)
  • Hebrew-first RTL with full i18n framework
  • 0% commission — attractive for price-sensitive market

Recommendation

Ready for closed beta with a tight-knit community (neighborhood group, Facebook community) where trust exists. Not ready for broad market launch without addressing payment, monetization, and growth mechanics.

Data Model & State Machine

Schema Alignment & Transition Audit

Delivery Status State Machine

NEW → PENDING → WAITING → PICKED_UP → DELIVERED → COMPLETED_PAID
NEW → CANCELLED
PENDING → NEW (driver cancels / 72h timeout)
PENDING → CANCELLED
WAITING → CANCELLED (pre-pickup)
PICKED_UP → NO cancellation allowed

Deviations from SOW

  • Timeout 24h vs 72h — code uses 24h, SOW says 72h
  • Cancel doesn't return to feed — SOW says listing returns as "new", code sets terminal "cancelled"
  • Extra transitions — new→waiting (multi-driver bypass), waiting→new (cancel selection)

✓All HLD fields present
