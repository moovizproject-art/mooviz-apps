This report presents the findings of a comprehensive security audit conducted on the MOOVIZ delivery platform. A total of 23 security findings were identified across all layers of the application stack. Of these, 22 have been fully remediated and 1 has been accepted with appropriate risk mitigation controls in place. The overall risk posture has been reduced from HIGH to LOW, and the platform is considered suitable for production deployment.

2 Scope

The following components were included in the security assessment:

  • React Native Mobile Application — iOS and Android builds (Expo SDK 50+, React Native 0.75.5)
  • Firebase Cloud Functions — Node.js 18, Gen 2 serverless functions (callable + triggers + scheduled)
  • Firestore Security Rules — Role-based access control for all collections
  • Firebase Storage Rules — Access control for media uploads (profile photos, delivery proofs, chat attachments)
  • Admin Panel — React Web application (Vite) for platform administration
  • Authentication System — Firebase Auth with OTP (SMS) and Email sign-in methods
  • Encryption Subsystem — AES-256-CBC encryption for sensitive data at rest

3 Methodology

The audit was performed using a combination of automated and manual techniques aligned with industry standards:

  • Static Application Security Testing (SAST) — Automated code scanning for known vulnerability patterns, insecure configurations, and hardcoded secrets across the entire codebase
  • Manual Code Review — Line-by-line review of security-critical modules including authentication flows, status transition logic, encryption routines, and access control enforcement
  • Configuration Review — Analysis of Firebase project configuration, security rules, CORS policies, Cloud Function settings, and environment variable management
  • OWASP Top 10 Assessment — Systematic evaluation against the OWASP Top 10 (2021) categories: Broken Access Control, Cryptographic Failures, Injection, Insecure Design, Security Misconfiguration, Vulnerable Components, Authentication Failures, Data Integrity Failures, Logging Failures, and SSRF
  • Firebase-Specific Security Checklist — Review of Firestore rules syntax and coverage, Storage rules participant verification, Cloud Function authentication enforcement, Secret Manager usage, and API key restrictions

5 Security Controls — Tests Passed

The following security controls were verified as correctly implemented and operational:

  • ✓Firebase Authentication required for all API calls
  • ✓Server-side validation for all delivery status transitions
  • ✓AES-256-CBC encryption with server-only key management
  • ✓RBAC enforcement across Firestore, Storage, and Cloud Functions
  • ✓Rate limiting on sensitive operations (OTP, login, interest, email)
  • ✓Input sanitization on email HTML content
  • ✓CORS restricted to authorized admin domains
  • ✓No PII in application logs
  • ✓Secrets managed via Firebase Secret Manager
  • ✓Storage rules enforce participant-based access
  • ✓Field-level write restrictions on delivery documents
  • ✓Role validation prevents privilege escalation on registration
  • ✓Proof photo system with mandatory uploads
  • ✓Encrypted profile photos with authorized access only
  • ✓Admin actions audit logging
  • ✓No hardcoded credentials in source control

6 Recommendations

The following recommendations are provided to further strengthen the security posture of the MOOVIZ platform:

  1. Deploy to production Firebase project with separate credentials — Ensure complete environment isolation between development, staging, and production with distinct API keys, service accounts, and database instances.
  2. Enable Firebase App Check for API abuse prevention — Implement App Check with DeviceCheck (iOS) and Play Integrity (Android) attestation providers to prevent unauthorized API access from non-genuine app instances.
  3. Implement Content Security Policy headers for admin panel — Add strict CSP headers to the admin web application to mitigate XSS and data injection attacks.
  4. Add automated SAST to CI/CD pipeline — Integrate static analysis tools (e.g., Snyk, Semgrep) into the EAS Build and Firebase deploy pipelines for continuous vulnerability scanning.
  5. Schedule quarterly security reviews — Establish a recurring security review cadence to assess new features, dependency updates, and evolving threat landscape.
  6. Consider WAF for Cloud Functions HTTP endpoints — Evaluate Cloud Armor or equivalent WAF solution for HTTP-triggered Cloud Functions to provide DDoS protection and request filtering.
  7. Implement session timeout for admin panel — Add configurable idle session timeout (recommended: 30 minutes) with automatic logout and re-authentication requirement.

7 Conclusion

The MOOVIZ platform has undergone a thorough security assessment covering all critical application layers. The audit identified 23 findings spanning Critical, High, Medium, and Low severity levels.

Through a focused remediation effort, 22 of 23 findings have been resolved. The single accepted risk (C4 — admin panel direct Firestore access) is appropriately mitigated by RBAC enforcement at the database security rules layer and is consistent with standard Firebase admin patterns.

The overall risk rating has been reduced from HIGH to LOW. All critical vulnerabilities — including client-side status transition bypasses, exposed encryption keys, and hardcoded secrets — have been fully remediated with server-side enforcement.

The system is suitable for production deployment with the noted accepted risks and the implementation of the recommendations outlined in this report.
